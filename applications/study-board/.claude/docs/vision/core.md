# Vision — study-board

## なぜ作るか

「書籍やチュートリアルで学んだはずの JS / React の基本が、実装する段になると毎回詰まる」を解消する。原因は **手を動かす量が足りない** + **詰まりポイントが言語化されていない** こと。

このプロジェクトでは：

1. Trello 風 Kanban を React で実装しながら基本構文・配列メソッド・イベント・state を体得する
2. **詰まったらアプリ上でカードに記録 → 解決時に自分の言葉でメモ欄に書く**
3. つまり **アプリが学習対象 + 学習を支える道具** の両方を兼ねる（ドッグフード）

## 学習ゴール（自分の言葉で説明できれば達成）

- 配列メソッド系: `map` / `filter` / `reduce` / `forEach` / `find` / `some` の選び方
- イベント: `onClick={fn}` vs `onClick={() => fn(arg)}` の違いと罠
- state: `useState` の更新関数で前の値を使う書き方、参照渡しの罠
- アロー関数: `this` の違い、`return` 省略、`{}` を返す時のカッコ
- 比較・型: `===` の理由、`null` vs `undefined`

詳細トピック → `../requirements/learning-roadmap.md`

## 設計判断

| 項目         | 選択                                                | 理由                                                                    |
| ------------ | --------------------------------------------------- | ----------------------------------------------------------------------- |
| 言語         | 素の JavaScript                                     | TypeScript は型レイヤーで学習を二重にする。先に JS を体得してから後付け |
| Style        | Vanilla CSS                                         | Tailwind / CSS-in-JS はそれ自体の学習コストが高い                       |
| State 管理   | `useState` のみ                                     | Redux / Zustand は不要、Context すら最初は使わない                      |
| 永続化       | Phase 2 で `localStorage`                           | DB / Server を絡めると関心事が散らかる                                  |
| テスト       | しない                                              | TDD / テスト設計は別ジャンル（`testing-and-quality/`）で扱う            |
| ディレクトリ | `applications/study-board/` 直下にコード + 教材同居 | `_spikes/` 分離方式は本プロジェクトでは認知負荷が勝る（規約改訂）       |

## non-Goals

- 美しい UI（CSS は機能が動くまで最小限）
- 本番デプロイ（学習が目的、配布しない）
- マルチユーザ / Cloud 同期（将来別プロジェクトで）
- アクセシビリティの完全準拠（後で `ui-rendering/` ジャンルで掘る）

## 将来構想: MCP 統合（Phase 4 候補）

学習が進んだら、Kanban のデータを **Claude が直接操作できる MCP サーバ** を Node.js で書く。

- Claude が「このカードを学習中に動かして」と言われたら自分で動かす
- カードのメモ欄を Claude と一緒に編集
- これが **学習者 + Claude ペア** の完成形

ただしこれは MCP プロトコル自体の学習になるため、Phase 1-3 完了後の選択肢として保留。MCP の概念学習は将来 `network-and-async/` または独立トピックとして扱う可能性がある。
