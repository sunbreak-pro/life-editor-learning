# Data Modeling — 概要（Overview）

> このジャンルの **入口**。なぜ「行と列の表」でデータを表現するのか、それは何を解決するために生まれたのかを、歴史と動機から押さえる。

---

## 1. そもそもの問い

「現実世界の何かをコンピュータに覚えさせる」ためには、形が必要。形の選び方は無数にある:

- メモ帳に文を書く
- フォルダにファイルを置く
- スプレッドシートに表として書く
- グラフ（点と線）として書く
- 時系列のログとして書く

このうちの 1 つが「**行と列の表 = リレーショナルテーブル**」。なぜこれが「データベースの主役」になったのか？

---

## 2. リレーショナルモデルの誕生（1970 年）

提唱したのは IBM 研究者の **E. F. Codd**。当時の主流は階層型 DB（IMS）やネットワーク型 DB だった。これらは:

- データの **物理的な格納構造** をアプリケーションコードが知っている必要があった
- 「商品データを別ディスクに引っ越す」となるとアプリ側も書き換え必須
- クエリは「ポインタを辿る」命令的なコード

Codd はこれに対し、**3 つの問題** を同時に解こうとした:

### 2.1 データ独立性 (Data Independence)

> 「データの **物理的な置き方** を変えても、**論理的なクエリ** は変えなくて済むようにしたい」

例: テーブルに新しいインデックスを足しても、`SELECT * FROM users WHERE name = 'taro'` の書き方は変わらない。
当時はこれが革命的だった。

### 2.2 整合性 (Integrity / 同じ事実を 2 箇所に書かない)

> 「**同じ情報を複数箇所に持つと、必ず食い違う**。だから 1 箇所に集約する仕組みが要る」

例: 顧客の住所をオーダーテーブルにも書いておくと、引っ越したとき両方更新しないと矛盾する。
これが **正規化（normalization）** の出発点。`(2) 重複排除` という直感はここに到達している。

### 2.3 宣言的クエリ (Declarative Query)

> 「**何が欲しいか** だけ書けば、**どう取ってくるか** はマシンに任せる」

階層型 DB のクエリ:

```
ROOT を開く → 子ノードを辿る → 名前 'taro' を探す → 見つかった行を返す
```

リレーショナルクエリ:

```sql
SELECT * FROM users WHERE name = 'taro';
```

「どこをどう辿るか」は **クエリオプティマイザ** が決める。人間は「何が欲しいか」だけを宣言する。
これは **集合論** に基づくモデル（行の集合 = 関係）として可能になった。

---

## 3. ユーザーの直感を 3 つの動機に位置づける

最初の想起で挙がった 3 つの観点を、Codd の動機と照らすと:

| ユーザーの選択                 | Codd の動機との関係                                                                             |
| ------------------------------ | ----------------------------------------------------------------------------------------------- |
| (2) 重複を 2 箇所に書かない    | **核心。整合性 §2.2 そのもの**。これが正規化につながる                                          |
| (1) コンピュータが計算しやすい | **結果論かつ部分真実**。実は RDB は **CPU 効率を犠牲にして** 整合性と人間の便利さを優先している |
| (4) ファイルより速い           | **結果論**。インデックス + クエリ最適化 + キャッシュのおかげ。テーブルそのものは遅い            |

### 3.1 「(1)(4) が逆」は何を意味するか

1970 年代、ファイル直書きや階層型 DB は **読み書きが速かった**。  
リレーショナル DB は遅かった。それでも普及したのは、**速さより整合性と独立性のほうが価値があった** から。

つまり、現在「RDB は速い」と感じるのは、**40 年かけてエンジニアたちが頑張って速くした** 結果であって、本質は依然として「(2) 整合性」が core。

これを掴むと、後で出てくる **NoSQL** の存在意義（「整合性をある程度諦めて速度を取り戻す」）が腑に落ちる。

## 4. 行・列・関係（用語の最小セット）

ここで導入する用語は **5 個** だけ（ワーキングメモリを尊重）:

| 用語                     | 意味                                                 |
| ------------------------ | ---------------------------------------------------- |
| **関係 (Relation)**      | テーブル本体。1 つのテーブル = 1 つの関係            |
| **タプル (Tuple)**       | 1 行 = 1 つのタプル                                  |
| **属性 (Attribute)**     | 1 列 = 1 つの属性                                    |
| **主キー (Primary Key)** | タプルを **一意に特定する** 属性（または属性の組）   |
| **整合性 (Integrity)**   | データに矛盾がない状態。RDB が一番大事にしているもの |

> 残りの用語（外部キー / 正規形 / カーディナリティ / インデックス / スキーマ など）は **次のセッション** で順に導入する。今ここで全部覚えなくて良い。

## 7. 補足: なぜこの順序で学ぶのか

「テーブルとは → 主キーとは → 外部キーとは → 正規化 → ER 図」と **下から積み上げる** 教え方が多いが、本ファイルは逆向きに **「なぜリレーショナルモデルが生まれたか」** から始めた。

理由:

- 用語を先に覚えても、**何を解いているか** が見えないと記憶が定着しない（学習科学 原則 7: 概念 → 実装 → 比較）
- 「整合性のために生まれた」と最初に分かれば、後から出てくる正規化・外部キー・トランザクションが **同じ物語の続き** として見える
- 結果として、別 DB（MongoDB / Neo4j / Datomic 等）に移ったときも「どの動機を諦めて、どの動機を強化したか」で位置づけられる
